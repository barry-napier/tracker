# E2E-1 Plan: Machine-readable widget list

## Steps

1. **Create `widgets.json` at repo root.** Content is the JSON array of non-empty
   lines of `widget.txt`, in file order. For the current input that is exactly:
   ```json
   ["the widget"]
   ```
   Trailing newline at EOF (matches the other two files). Derive it by reading
   `widget.txt` sequentially and skipping blank lines — do not hand-type from
   memory and do not sort.

2. **Edit `README.md`.** Keep the existing `# gh-proof scratch` heading; add a
   short section (e.g. `## widgets.json`) stating:
   - what it is: a JSON array of the widget names (strings);
   - provenance: derived from the non-empty lines of `widget.txt`, same order,
     and must be regenerated by hand if `widget.txt` changes (no generator
     script exists — ticket doesn't ask for one, don't add one).

3. **Verify** by running the checks below from the repo root.

## Verification seams

Both ACs are machine-checkable; no human routing needed.

- `checks/ac-1.sh` (AC-1): asserts `widgets.json` exists at repo root, parses as
  JSON, top-level value is an array of strings, and the entries equal the
  non-empty lines of `widget.txt` in order (computed from `widget.txt` at check
  time, so the check stays valid if the input file ever changes). Uses python3
  (stdlib json) — no repo tooling exists, and python3 is on PATH.
- `checks/ac-2.sh` (AC-2): asserts `README.md` mentions `widgets.json`,
  mentions `widget.txt`, describes the contents (matches "widget name" or
  "array"), and uses derivation wording (deriv/generat/based on/from
  widget.txt/source). Textual heuristics — writing the README section per step 2
  satisfies them naturally.
- `checks/manifest.json` maps AC 1 → `checks/ac-1.sh`, AC 2 → `checks/ac-2.sh`.

Both scripts `cd` to the repo root relative to their own location, so they can
be invoked from anywhere. Confirmed behavior: both fail on the current tree
(files not yet created/edited) and pass on a simulated completed state.

## Constraints / non-goals

- No generator script, no CI, no package manifest — out of scope per research
