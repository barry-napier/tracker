# Frozen ac-checks: the implementing session never writes its own exam

Until TRK-2, ac-check scripts were authored by the plan phase and executed by the battery — but nothing stopped the implement session (same worktree, later in the same Run) from quietly rewriting a script it was about to be graded by. The trust boundary said "results cannot be self-reported"; the exam itself was still forgeable.

Now the exam freezes at authoring time. `readCheckManifest` fingerprints every script's exact bytes (sha256) the moment a manifest validates — at the check-emitting node's exit, before any implementing session runs — and `registerAcChecks` records the hash on the `ac_checks` row. The battery's ac-check gate re-hashes the worktree file before executing it: any drift is a fail on its own terms (the script never runs, so a weakened exam grades nothing), carrying frozen and actual hashes in the gate detail. Authoring sessions may re-freeze — a later Run's plan phase re-registering pending ACs records new hashes — because plan is not the session being examined; the freeze holds from registration to execution within a cycle.

Follow-up Criteria get the same treatment at bounce time (AC-37): `CheckAuthor` runs a dedicated one-shot provider session — never the future implementing session — that converts the freshly minted follow-ups into scripts or human routings, validated and frozen through the same manifest path. To make that raceless, both bounce paths mint follow-up AC rows *before* the state move (`mintFollowUpAcs`): the ticket still sits in Verifying/Human Review, states no worker claims, so the authoring session has the worktree to itself and a re-claim can never start against check-less follow-ups. Authoring is best-effort by design: a crashed or non-covering session is reported loudly and the bounce proceeds — the next plan phase covers the gap, which is exactly the pre-TRK-2 behavior. A bounce about to park skips authoring (the next session is a human, not a run); the review-fail verdict request waits on the authoring session — a stated trade, visible in the run log as `author-checks`.

Considered: storing script blobs and executing the stored copy (rejected — AC-36's own wording is hash-verification, and a second copy invites divergence between what's shown and what runs); file-permission enforcement (rejected — worktrees are the agent's own sandbox, chmod is theater against a session with shell access; detection at the gate is the teeth); authoring at intake approval (deferred — the ticket allows it, but plan-exit freezing already puts authoring in a separate session from implement without new intake machinery); background authoring after the state move (rejected — races the re-claim for the worktree and can strand a new Run's plan phase mid-write).
