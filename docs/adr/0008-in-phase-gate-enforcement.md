# In-phase gate enforcement: a cheap subset at every phase exit

Verification used to run exactly once, at Verifying (ADR-0003): a trivially lintable defect — a recap without review notes, a red suite — cost the full remaining graph walk, the demo, the whole battery, and a bounce cycle before anyone said so. TRK-1 moves a cheap subset of that judgment to every `agent_phase` exit: the engine runs node-scoped artifact lint (only what *this* node's `gate_requirements` declare, so research never dies on a recap the document phase hasn't written) plus the repo suite, and a contract-clean exit is only valid once they pass. On failure the findings re-prompt the SAME live provider session (`RunPhaseOpts.resumeSessionId`, capability-gated by `supportsResume`; a provider without resume gets a standalone re-brief), capped at 2 retries per phase attempt. Exhaustion is a phase death (`gate-exhausted`) riding the existing once-retry/crash/cap policy — worst case 6 provider invocations per phase.

This amends ADR-0003's timing, never its trust model: the orchestrator still executes every check, nodes still only declare what they owe, and the full battery at Verifying is unchanged — greenness (dogfood-green), freshness (pr-fresh, demo-fresh), and AC checks remain the battery's alone. In-phase rows land in `gate_results` and the audit trail like any gate, wearing the `phase-gate:` prefix so the bounce machinery and the wizard's badge chrome (battery readers both) can ignore them — an in-phase failure the agent then fixed is history, not a defect, and must not resurrect as a follow-up AC.

One semantic shift, endorsed by the ticket's ACs: a defect the agent cannot fix in-phase (a persistently red suite) now reads as crash-flavored (phase death → Todo → crash cap) rather than bounce-flavored, because the work never gets far enough for the battery to call it wrong.

Considered: verification nodes in the workflow graph (rejected by ADR-0003 — users customize how work gets done, never whether it gets checked); running the full battery per phase (rejected — dogfood-green/pr-fresh/demo-fresh judge run-level outcomes that don't exist mid-graph, and the demo alone would dwarf the phases); a fresh run per gate failure (rejected — that is the bounce cycle this exists to avoid; the live session already holds the context the fix needs).
